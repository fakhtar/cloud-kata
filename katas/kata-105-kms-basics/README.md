# kata-105 — KMS Basics: Keys, Aliases & Rotation

## Overview

In this kata you will create a customer managed KMS key configured for
symmetric encryption, assign it a human-readable alias, and enable automatic
key rotation. You will verify that the key is correctly configured, enabled,
and ready for use.

This kata focuses on key configuration — not on encrypting or decrypting
data. Understanding how to create, alias, and rotate KMS keys is foundational
to any AWS workload that handles sensitive data.

---

## Prerequisites

- An active AWS account
- Access to AWS CloudShell
- IAM permissions to create and manage KMS keys and aliases
- Familiarity with basic encryption concepts

---

### Requirement 1 — Customer Managed Key

Create a KMS key with the following configuration:

- **Key type:** Symmetric
- **Key usage:** Encrypt and decrypt
- **Origin:** AWS KMS (key material generated by AWS)
- The key must be in an **enabled** state

---

### Requirement 2 — Key Alias

Create an alias for the key:

- **Alias name:** `alias/kata-105-key`
- The alias must point to the key created in Requirement 1

---

### Requirement 3 — Key Rotation

Enable automatic key rotation on the key. Key rotation is a security best
practice that ensures the key material is periodically refreshed without
disrupting existing encrypted data.

> ℹ️ Automatic key rotation is only supported on symmetric encryption keys
> with AWS-generated key material. This is the key type required in
> Requirement 1.

## Cleanup

Always clean up your resources when you are finished.

### Step 1 — Delete CloudFormation stacks (if applicable)

If you deployed `solution.yml`, delete that stack first via CloudFormation.

- Go to the AWS CloudFormation console
- Select the stack created for this kata
- Choose **Delete** and wait for deletion to complete

Via CLI:
```bash
aws cloudformation delete-stack --stack-name kata-105-solution
aws cloudformation wait stack-delete-complete --stack-name kata-105-solution
```

> ⚠️ Deleting the CFN stack schedules the KMS key for deletion with a
> waiting period. The key will not be immediately deleted — it enters a
> pending deletion state for the configured number of days. You will
> continue to be charged $1.00/month during this waiting period.

### Step 2 — Manual Cleanup

If you created resources manually:

1. Delete the alias `alias/kata-105-key` in the KMS console
2. Schedule the key for deletion (minimum 7 day waiting period)

Via CLI:
```bash
# Delete the alias
aws kms delete-alias --alias-name alias/kata-105-key

# Schedule key deletion (replace KEY_ID with your key ID)
aws kms schedule-key-deletion \
  --key-id <KEY_ID> \
  --pending-window-in-days 7
```

### Step 3 — Verify in the console

Go to the KMS console and confirm the alias no longer exists and the key
is in **Pending deletion** state.
