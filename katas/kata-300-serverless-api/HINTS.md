## Requirement 1 — DynamoDB Table

<details>
<summary>Hint 1 — How DynamoDB identifies items</summary>

DynamoDB is a key-value store. Every item in a table must have a value for its
primary key, which DynamoDB uses to locate the item. The simplest form of
primary key is a single attribute — called the partition key — that must be
unique across all items. Think about what attribute from the function code
would serve as that unique identifier.

</details>

<details>
<summary>Hint 2 — Choosing the key type and data type</summary>

Look at the function code. Items are stored with an `id` field whose value is
generated by `uuid.uuid4()`. UUIDs are strings. DynamoDB requires you to declare
key attributes upfront in the table definition with both their name and type
(`S` for String, `N` for Number, `B` for Binary). You only need to declare
attributes that are part of the key schema — not every field your items will have.

</details>

<details>
<summary>Hint 3 — Exact configuration</summary>

- **Table name:** `kata-300-ItemsTable`
- **Partition key:** attribute name `id`, type `String`
- **No sort key** is needed — `id` alone is sufficient as a simple primary key
- **Capacity mode:** on-demand (PAY_PER_REQUEST) avoids the need to set read
  and write capacity units

In the DynamoDB console, choose **Create table**, set the table name, and under
**Partition key** enter `id` with type `String`. Leave **Sort key** empty.
Under **Table settings**, choose **Customize settings** and set
**Read/write capacity settings** to **On-demand**.

</details>

## Requirement 2 — Lambda Function

<details>
<summary>Hint 1 — What permissions the function needs</summary>

The function has two jobs: write log output (all Lambda functions do this) and
read and write items in a DynamoDB table. Both require explicit permissions via
an IAM execution role. Think about which AWS services are involved and what
actions the function code actually performs — `put_item` and `scan` map directly
to IAM actions.

</details>

<details>
<summary>Hint 2 — Passing the table name to the function</summary>

The function code reads the table name from `os.environ["TABLE_NAME"]` at
startup. This means the table name must exist as an environment variable named
exactly `TABLE_NAME` in the function's configuration. Lambda environment
variables are key-value string pairs set in the function's configuration — not
in the code. The value should be the actual table name, not a hardcoded string
in the source.

</details>

<details>
<summary>Hint 3 — Exact configuration</summary>

- **Function name:** `kata-300-HandlerFunction`
- **Runtime:** Python 3.12
- **Handler:** `index.handler` (if using CloudFormation inline code, the file
  is automatically named `index.py`)
- **Execution role:** an IAM role with:
  - `AWSLambdaBasicExecutionRole` managed policy (CloudWatch Logs access)
  - An inline or attached policy allowing `dynamodb:PutItem` and `dynamodb:Scan`
    on the `kata-300-ItemsTable` ARN
- **Environment variable:** key `TABLE_NAME`, value `kata-300-ItemsTable`

In the Lambda console, set the environment variable under
**Configuration** → **Environment variables**. Grant DynamoDB permissions by
attaching an inline policy to the execution role with the two DynamoDB actions
scoped to the table's ARN.

</details>

## Requirement 3 — REST API

<details>
<summary>Hint 1 — REST API vs HTTP API</summary>

API Gateway offers two main API types: REST APIs (v1) and HTTP APIs (v2). They
look similar in the console but are different products with different feature
sets and CLI namespaces. This kata uses a REST API. When creating via the
console, choose **REST API** — not HTTP API, not WebSocket, and not the
**REST API (Private)** variant.

</details>

<details>
<summary>Hint 2 — Creating the API</summary>

In the API Gateway console, choose **Create API** then select **REST API**.
Choose **New API**, set the name to `kata-300-ItemsAPI`, and leave the endpoint
type as **Regional**. No other settings are required at creation time.

</details>

<details>
<summary>Hint 3 — Exact name</summary>

The API must be named exactly `kata-300-ItemsAPI`. The validator finds the API
by name using `aws apigateway get-rest-apis`, so capitalisation and hyphens
must match exactly.

</details>

## Requirement 4 — Resources and Methods

<details>
<summary>Hint 1 — How API Gateway models paths and verbs</summary>

In a REST API, a resource represents a URL path segment. A method sits on a
resource and handles a specific HTTP verb. To support two operations under the
same path, you create one resource and two methods on it — one for each
operation. The HTTP conventions for creating a new resource and listing existing
ones are well established.

</details>

<details>
<summary>Hint 2 — Choosing the right path and verbs</summary>

The function code routes on both `httpMethod` and `path`. It handles
`POST /items` for creating and `GET /items` for listing. The resource path
must be `/items` and you need both a `POST` method and a `GET` method on that
resource.

</details>

<details>
<summary>Hint 3 — Exact configuration</summary>

In the API Gateway console, select your API and go to **Resources**. Create a
new resource with path segment `items` under the root `/` — the resulting path
will be `/items`. Then create two methods on that resource: `POST` and `GET`.

If using CloudFormation, the `PathPart` property takes just the segment name
without the leading slash: `items`, not `/items`.

</details>

## Requirement 5 — Lambda Integration

<details>
<summary>Hint 1 — What proxy integration means</summary>

API Gateway supports two modes for Lambda integration. In a custom integration,
API Gateway transforms the request and response using mapping templates you
define. In a proxy integration, API Gateway forwards the entire HTTP request
to Lambda as a structured event object and returns the function's response
object directly to the caller — no transformation needed. The function is
responsible for returning a well-formed response with `statusCode`, `headers`,
and `body`. The function code in this kata is already written for proxy
integration.

</details>

<details>
<summary>Hint 2 — Wiring the integration and granting permission</summary>

When configuring each method, choose **Lambda function** as the integration
type and enable the **Lambda proxy integration** toggle. Point both methods at
`kata-300-HandlerFunction`. API Gateway will prompt you to add a
resource-based permission to the function so it can invoke it — accept this
prompt for each method, or add the permission manually.

If you skip the permission step, the integration will appear wired but
invocations will fail with a 500 error. The validator checks both that the
integration type is `AWS_PROXY` and that the endpoint returns the correct
status code.

</details>

<details>
<summary>Hint 3 — CloudFormation integration properties</summary>

Both methods need an identical `Integration` block:

```yaml
Integration:
  Type: AWS_PROXY
  IntegrationHttpMethod: POST
  Uri: !Sub
    - "arn:aws:apigateway:${AWS::Region}:lambda:path/2015-03-31/functions/${FunctionArn}/invocations"
    - FunctionArn: !GetAtt HandlerFunction.Arn
```

`IntegrationHttpMethod` must be `POST` for Lambda integrations — this is the
method API Gateway uses internally to call Lambda, regardless of the
client-facing HTTP method (GET, POST, or anything else).

A single `AWS::Lambda::Permission` resource with a wildcard `SourceArn`
covers both methods:

```yaml
LambdaInvokePermission:
  Type: AWS::Lambda::Permission
  Properties:
    FunctionName: !GetAtt HandlerFunction.Arn
    Action: lambda:InvokeFunction
    Principal: apigateway.amazonaws.com
    SourceArn: !Sub "arn:aws:execute-api:${AWS::Region}:${AWS::AccountId}:${ItemsApi}/*/*/*"
```

</details>

## Requirement 6 — Deployment and Stage

<details>
<summary>Hint 1 — Why deployment is a separate step</summary>

In API Gateway REST APIs, changes to resources and methods are not live until
you deploy them. A deployment is a snapshot of your API configuration at a
point in time. A stage is a named pointer to a deployment — it forms the URL
prefix that callers use. Until you deploy, the endpoint does not exist and
cannot be invoked.

</details>

<details>
<summary>Hint 2 — Deploying in the console</summary>

In the API Gateway console, go to your API and choose **Deploy API**. When
prompted, create a new stage named `prod`. After deployment, the invoke URL
will look like:
`https://{api-id}.execute-api.{region}.amazonaws.com/prod`

Append `/items` to test. Any time you make changes to methods or integrations
you must redeploy for the changes to take effect.

</details>

<details>
<summary>Hint 3 — CloudFormation deployment and the DependsOn requirement</summary>

Use `AWS::ApiGateway::Deployment` with `StageName: prod`. Because this kata
has two methods, the deployment must depend on both:

```yaml
ApiDeployment:
  Type: AWS::ApiGateway::Deployment
  DependsOn:
    - ItemsPostMethod
    - ItemsGetMethod
  Properties:
    RestApiId: !Ref ItemsApi
    StageName: prod
```

Without `DependsOn`, CloudFormation may create the deployment before either
method exists and fail with: `The REST API doesn't contain any methods`.

</details>

## Requirement 7 — Tags

<details>
<summary>Hint 1 — Three resources need tags</summary>

This kata requires tags on three separate resources: the DynamoDB table, the
Lambda function, and the REST API. Missing tags on any one of them will cause
the corresponding validator check to fail. All three must carry the same two
tags.

</details>

<details>
<summary>Hint 2 — Where to add tags in the console</summary>

For the **DynamoDB table**: go to the table, choose the **Additional settings**
tab, and find the **Tags** section at the bottom.

For the **Lambda function**: go to the function page, choose
**Configuration** → **Tags** → **Manage tags**.

For the **REST API**: go to the API in the API Gateway console, choose
**Settings** at the API level (not the stage level) and find the **Tags**
section.

</details>

<details>
<summary>Hint 3 — Exact tag values</summary>

Tags are case-sensitive. The validator checks for these exact values on all
three resources:

- Key: `Project` — Value: `CloudKata`
- Key: `Kata` — Value: `kata-300`

</details>

## General Troubleshooting

<details>
<summary>Check 3 failing — TABLE_NAME environment variable not found</summary>

The validator reads the function's environment variables using
`aws lambda get-function-configuration` and checks that `TABLE_NAME` equals
`kata-300-ItemsTable`. If this check fails, either the variable is missing
entirely or the value doesn't match the expected table name exactly. In the
Lambda console, go to **Configuration** → **Environment variables** and verify
the key is `TABLE_NAME` (all caps) and the value is `kata-300-ItemsTable`.

</details>

<details>
<summary>Check 8 or 9 failing — integration type is not AWS_PROXY</summary>

The validator reads `.methodIntegration.type` from `get-method` and checks for
`AWS_PROXY`. If you configured the method without enabling the proxy integration
toggle, the type will be `AWS` instead. Delete the method and recreate it with
proxy integration enabled, or change the integration type via
**Method Execution** → **Integration Request** in the console.

</details>

<details>
<summary>Check 11 or 12 failing — endpoint returns 500 or unexpected status</summary>

A 500 response almost always means one of three things. First, the Lambda
permission is missing — API Gateway cannot invoke the function. Go to the
Lambda function, choose **Configuration** → **Permissions**, and verify there
is a resource-based policy allowing `apigateway.amazonaws.com` to invoke it.
Second, the function is failing at runtime — check CloudWatch Logs for the
function to see the error. The most common runtime failure is a missing or
incorrect `TABLE_NAME` environment variable, or insufficient DynamoDB
permissions on the execution role. Third, the API has not been redeployed after
a change — any modification to a method requires a new deployment before it
takes effect in the `prod` stage.

</details>

<details>
<summary>Check 11 failing specifically — POST returns 404 instead of 201</summary>

The function routes on both `httpMethod` and `path`. If the path the function
receives does not match `/items` exactly, it will fall through to the 404
branch. This happens when the API resource is misconfigured — for example, if
the path part was set to `/items` (with a leading slash) instead of `items`.
Verify the resource path in API Gateway is `/items` and redeploy.

</details>

<details>
<summary>CloudFormation DependsOn error — REST API has no methods</summary>

If your stack fails with `The REST API doesn't contain any methods`, the
`AWS::ApiGateway::Deployment` resource was created before the methods were
ready. Add `DependsOn` listing both method logical IDs to the deployment
resource. CloudFormation does not infer this dependency automatically because
the deployment only references the API ID, not the individual methods.

</details>

## Still stuck?

The complete working solution is in [solution.yml](./solution.yml).

Deploy it with:

```bash
aws cloudformation deploy \
  --template-file solution.yml \
  --stack-name kata-300-solution \
  --capabilities CAPABILITY_NAMED_IAM
```

The `--capabilities CAPABILITY_NAMED_IAM` flag is required because the template
creates an IAM role with an explicit name.

Then re-run `validate.sh`. A correctly deployed solution should score 15/15.
Study the solution to understand what you missed, then tear it down and try
again from scratch.
